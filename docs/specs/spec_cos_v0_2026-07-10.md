## Intent

This is the v0 build of the Chief of Staff capability — not a POC, not a throwaway. It is a case study that proves the architecture and justifies the direction. Every design decision is made with the production system in mind.

The goal: Robert talks naturally to the minimoi Telegram bot. CoS listens, responds as a thinking partner, and stores what matters to a structured memory store — without Robert ever thinking about the plumbing.

This replaces the manual decisions.md process that was designed but never adopted because it required deliberate action. The new model is fluid and natural — conversation happens, memory accumulates automatically.

---

## The Experience

Robert opens Telegram and talks to `@minimoi_agent_bot` — the same bot he already uses for German and Curator commands. No new app, no new interface, no prefix to remember.

```
Robert: "I'm thinking the Curator Tech domain should start 
         inside Curator, not as a standalone domain"

CoS: "That makes sense — inherits the scoring engine and 
      pipeline immediately. Worth noting as a direction 
      decision. Anything specific driving the timeline?"

Robert: [voice note] "Also remind me to follow up with 
         Paul at Bennett Moore next week"

CoS: "Noted — follow up with Paul at Bennett Moore. 
      I'll flag it when next week starts."
```

That's it. Natural conversation. CoS handles the rest.

## Routing Architecture

All messages arrive on `@minimoi_agent_bot`. Routing logic:

```
Message arrives
  ├── !german  → German domain handler
  ├── !curator → Curator domain handler
  ├── !cos     → CoS explicit override (verification/testing only)
  └── Everything else → CoS by default
```

CoS is the default conversational layer. Domain commands are the exception. Robert never needs to think about routing.

---

## Two OpenClaw Instances — Hard Isolation

Personal OpenClaw (#1) and mini-moi OpenClaw (#2) are completely isolated. No shared memory, no shared filesystem, no overlap possible.

| | Personal OpenClaw (#1) | mini-moi CoS (#2) |
|---|---|---|
| Purpose | Personal life, unrestricted | mini-moi domains only |
| Memory | `~/.openclaw/` | `~/.openclaw-cos/` (container volume) |
| Port | 18789 | 18889 |
| Telegram | Personal bot | `@minimoi_agent_bot` default |
| Container | Runs as-is | New Docker container |
| Scope | Everything | mini-moi only, enforced by container |

Container isolation makes the scope boundary physically enforced — not just a config setting.

## Memory Store — JSON First, Postgres Schema Ready

**File:** `openclaw/cos_memory.json`
Volume-mounted on host — not in container, not in git. Persists across redeploys.

**Schema v1.0:**
```json
{
  "version": "1.0",
  "entries": [
    {
      "id": "uuid-v4",
      "timestamp": "2026-07-10T14:23:00Z",
      "actor": "robert",
      "channel": "telegram_text",
      "type": "note",
      "domain": "cross-domain",
      "content": "raw message or transcript",
      "summary": "CoS-generated one liner",
      "tags": ["minimoi", "curator-tech"],
      "linked_to": [],
      "stored_by": "cos-v0"
    }
  ]
}
```

**Entry types:**
- `note` — something worth remembering, Robert said it
- `decision` — a direction was set
- `action` — something to do or follow up on
- `question` — open question, no answer yet
- `observation` — CoS noticed something worth flagging
- `conversation` — raw exchange, no specific extraction

**What gets stored vs what doesn't:**
CoS decides. Not every message is stored. Casual exchanges, simple questions with immediate answers, and routine command routing are not stored. Decisions, meaningful notes, action items, and open questions are stored.

**Postgres schema (ready when needed — no rework):**
```sql
CREATE TABLE cos_memory (
  id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  timestamp TIMESTAMPTZ NOT NULL DEFAULT NOW(),
  actor VARCHAR(50) NOT NULL,
  channel VARCHAR(50),
  type VARCHAR(50) NOT NULL,
  domain VARCHAR(50) DEFAULT 'cross-domain',
  content TEXT NOT NULL,
  summary TEXT,
  tags JSONB DEFAULT '[]',
  linked_to JSONB DEFAULT '[]',
  stored_by VARCHAR(50),
  created_at TIMESTAMPTZ DEFAULT NOW()
);

CREATE INDEX idx_cos_memory_timestamp ON cos_memory(timestamp DESC);
CREATE INDEX idx_cos_memory_type ON cos_memory(type);
CREATE INDEX idx_cos_memory_domain ON cos_memory(domain);
CREATE INDEX idx_cos_memory_tags ON cos_memory USING GIN(tags);
```

JSON → Postgres migration: direct field mapping, no restructuring needed. Load script reads `cos_memory.json`, inserts all entries, done.

## Containerization

```yaml
# docker-compose additions
openclaw-cos:
  build:
    context: .
    dockerfile: docker/Dockerfile.openclaw-cos
  ports:
    - "18889:18889"
  volumes:
    - /opt/minimoi/openclaw:/app/openclaw          # shared memory store
    - /opt/minimoi/.openclaw-cos:/root/.openclaw    # OpenClaw internal memory
    - /opt/minimoi/docs:/app/docs:ro                # read-only docs
    - /opt/minimoi/build_queue.json:/app/build_queue.json:ro  # read-only queue
  env_file: /opt/minimoi/.env
  restart: unless-stopped
  networks:
    - minimoi-network
```

**What the container can access:**
- `/app/openclaw/` — write (memory store)
- `/app/docs/` — read only
- `/app/build_queue.json` — read only
- Telegram API (outbound)
- Other minimoi containers via Docker network (read health endpoints)

**What it cannot access:**
- Personal `~/.openclaw/` — not mounted, physically impossible
- Production docker socket (Phase 1 limit)
- Domain data files directly

## Dev Setup — Mac First

Run container on Mac dev. Same image that promotes to EC2.

```bash
# Mac dev startup
docker-compose -f docker-compose.dev.yml up openclaw-cos

# Telegram webhook during dev (temporary)
ngrok http 18889
# Set webhook URL in Telegram bot config to ngrok URL
```

**Dev → EC2 promotion:**
Same docker-compose pattern as every other service. No changes to container or config — just a different host. Connectivity setup (Telegram webhook URL → EC2 IP) is the only variable.

## First Use Case — The Complete Loop

**Input:** Robert sends any unstructured message or voice note to `@minimoi_agent_bot`

**Processing:**
1. Bot receives message
2. Routes to CoS (default handler)
3. If voice: Whisper transcription → text
4. CoS processes with OpenClaw + LLM context
5. CoS decides: store or not store
6. If storing: generate summary, classify type, tag domain
7. Append entry to `cos_memory.json`
8. Respond to Robert naturally

**Output:** Natural conversational response + silent memory write

Robert never sees the memory write. It just happens.

## Scope Enforcement

CoS knows it is scoped to mini-moi domains. If Robert asks about something outside scope:

```
Robert: "What should I make for dinner?"
CoS: "That's outside my scope here — I'm focused on 
      mini-moi and your work. Your personal OpenClaw 
      is better placed for that."
```

Scope defined in `config/cos_config.json`:
```json
{
  "scope": "mini-moi domains only",
  "allowed_domains": ["curator", "german", "portuguese", "guild", "career", "cross-domain"],
  "out_of_scope_response": "That's outside my scope here — I'm focused on mini-moi and your work.",
  "memory_store": "/app/openclaw/cos_memory.json",
  "model": "grok-4-1-fast-reasoning"
}
```

## What This Replaces

The manual decisions.md and actions.md process — designed but never adopted because it required deliberate action. This v0 replaces it with a fluid, natural process:

- **Before:** Robert had to remember to log a decision → friction → didn't happen
- **After:** Robert talks naturally → CoS extracts and stores what matters → no friction

The memory accumulates as a side effect of natural conversation. That's the right model.

## Case Study Framing

This v0 is designed as a case study to justify the broader CoS direction. Success criteria:

- Does natural conversation feel comfortable vs forced?
- Does CoS correctly identify what's worth storing?
- Does the memory store accumulate useful signal over 2-4 weeks?
- Does Robert refer back to stored memory (proving it's valuable)?
- Does the container isolation hold — no bleed between personal and mini-moi OpenClaw?

Document findings after 2-4 weeks of daily use. These findings feed directly into the Phase 1 full build spec and the agent evaluation (OpenClaw vs custom Grok agent).
